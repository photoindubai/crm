# Files and Cloudflare R2

## Why this layer exists

This CRM now has a dedicated files metadata layer to support:

- backward compatibility with legacy external URLs
- Cloudflare R2 as the default storage provider for new uploads
- future multi-tenant organization-aware storage paths

Physical objects live in R2, while file metadata is stored in Supabase (`public.files`).

## Buckets

- public bucket: `exhibition-public-assets`
- private bucket: `exhibition-private-files`

Use env values, not hardcoded bucket names in runtime logic.

## Required environment variables

- `R2_ACCOUNT_ID`
- `R2_ENDPOINT`
- `R2_ACCESS_KEY_ID`
- `R2_SECRET_ACCESS_KEY`
- `R2_PUBLIC_BUCKET`
- `R2_PRIVATE_BUCKET`
- `R2_PUBLIC_BASE_URL`
- `DEFAULT_ORGANIZATION_ID`
- `DEFAULT_ORGANIZATION_SLUG`

Important:
- keep `R2_ACCESS_KEY_ID` and `R2_SECRET_ACCESS_KEY` server-only
- do not hardcode `assets.chat-admin.online`
- always generate public links from `R2_PUBLIC_BASE_URL`

## Database model

Migration added `public.files` as the canonical metadata table.

Main fields:
- ownership context: `organization_id`, optional `event_id`, `company_id`, `participation_id`, `brand_id`
- target entity: `entity_type`, `entity_id`
- file role: `file_category`, `file_role`
- storage location: `provider`, `bucket`, `storage_path`
- URL fallback fields: `external_url`, `public_url`
- metadata: `original_filename`, `mime_type`, `size_bytes`
- lifecycle: `is_public`, `source`, `status`, `uploaded_by`, timestamps

Logo-specific convention:
- `file_category='logo'`
- `file_role` variants:
  - `full`
  - `thumb`
  - `full_inverted`
  - `thumb_inverted`
- legacy/backward-compatible roles still supported:
  - `primary`
  - `legacy_external`

Also added optional references:
- `companies.primary_logo_file_id`
- `brands.primary_logo_file_id`
- `participations.public_logo_file_id`

Legacy columns are intentionally preserved:
- `companies.company_logo_url`
- `brands.brand_logo_url`
- `exhibitor_materials.url`

## Public vs private files

Public files:
- `is_public = true`
- URL should be generated by `R2_PUBLIC_BASE_URL + "/" + storage_path`

Private files:
- `is_public = false`
- do not build URL from `R2_PUBLIC_BASE_URL`
- future step: signed private downloads

## R2 path conventions

Canonical prefix:

`organizations/{organization_slug}/...`

Use `DEFAULT_ORGANIZATION_SLUG` for current single-tenant MVP where org slug is not resolved from runtime tenant context.

Typical public paths:

- company logos/materials:
  - `organizations/{slug}/companies/{company_id}/logos/{file_role}/{file_id}-{safe_filename}`
  - `organizations/{slug}/companies/{company_id}/materials/{file_id}-{safe_filename}`
- brand logos:
  - `organizations/{slug}/brands/{brand_id}/logos/{file_role}/{file_id}-{safe_filename}`
- event assets:
  - `organizations/{slug}/events/{event_id}/floorplans/{file_id}-{safe_filename}`
  - `organizations/{slug}/events/{event_id}/public-documents/{file_id}-{safe_filename}`
- participation assets:
  - `organizations/{slug}/events/{event_id}/participations/{participation_id}/logos/{file_id}-{safe_filename}`
  - `organizations/{slug}/events/{event_id}/participations/{participation_id}/materials/{file_id}-{safe_filename}`

Filename rules:
- lowercase where practical
- replace spaces with hyphens
- remove unsafe characters
- prefix with `file_id` for collision safety

Logo thumbnail generation:
- Upload accepts full-size logo as source (`full` or `full_inverted`).
- For PNG/JPEG/WebP, server auto-generates 300x300 max thumbnail:
  - `full -> thumb`
  - `full_inverted -> thumb_inverted`
- No automatic color inversion is performed.
- Inverted logo must be uploaded explicitly as its own source file.
- SVG upload is accepted, but thumbnail generation can be skipped as fallback.

## Legacy URL behavior

Legacy URL fields remain valid and are not removed in this phase.

Expected read fallback order for UI and APIs:
1. `public.files` record (`provider + bucket + storage_path`, optionally `public_url`)
2. legacy direct URL fields (`company_logo_url`, `brand_logo_url`, `exhibitor_materials.url`)

No automatic download/copy of old external assets into R2 is performed in this phase.

## Phase 2 legacy metadata migration

Phase 2 registers existing legacy URLs into `public.files` metadata only.

Script:

- `scripts/migrate-legacy-files-metadata.mjs`

NPM command:

- `npm run migrate:legacy-files`

The npm script loads env from `.env`, then overrides from `.env.local` if present.

Dry run:

```bash
npm run migrate:legacy-files -- --dry-run
```

Apply:

```bash
npm run migrate:legacy-files
```

What the script does:
- registers `companies.company_logo_url` as `provider='external'`, `source='legacy_company_logo_url'`
- registers `brands.brand_logo_url` as `provider='external'`, `source='legacy_brand_logo_url'`
- registers `exhibitor_materials.url` as `provider='external'`, `source='legacy_exhibitor_materials_url'`
- populates `companies.primary_logo_file_id` and `brands.primary_logo_file_id` only when currently null
- keeps legacy URL columns unchanged
- does not upload anything to R2
- does not download or copy legacy files

Idempotency:
- duplicate prevention key for company logos:
  - `entity_type + entity_id + file_category + source + external_url`
- duplicate prevention key for brand logos:
  - `entity_type + entity_id + file_category + source + external_url`
- duplicate prevention key for materials:
  - `participation_id + source + external_url`

Script output counters:
- company logos registered
- brand logos registered
- exhibitor materials registered
- skipped duplicates
- errors

## Fallback resolver helpers

Helper files:

- [resolve.ts](/home/anton/projects/crm/src/lib/files/resolve.ts)
- [loaders.ts](/home/anton/projects/crm/src/lib/files/loaders.ts)

Resolver behavior:
- `getFilePublicUrl(file)`:
  - `cloudflare_r2`: build from `R2_PUBLIC_BASE_URL + storage_path` only for public files
  - `external`: use `public_url` or `external_url`
  - `supabase_storage/aws_s3`: use `public_url` when present
- `resolveCompanyLogo(...)`:
  1. `company.primary_logo_file`
  2. separately loaded `primaryLogoFile`
  3. `company.company_logo_url`
- `resolveBrandLogo(...)`:
  1. `brand.primary_logo_file`
  2. separately loaded `primaryLogoFile`
  3. `brand.brand_logo_url`
- `resolveParticipationLogo(...)`:
  1. `participation.public_logo_file`
  2. separately loaded `participationLogoFile`
  3. `company.primary_logo_file`
  4. separately loaded `companyLogoFile`
  5. `company.company_logo_url`

## Asset domain changes

Public domain changes should be handled by env update only:

- set new `R2_PUBLIC_BASE_URL`
- do not rewrite existing `storage_path`

Because `storage_path` is canonical, public URL generation can switch domains without data migration.

## Phase 3 upload API routes

Server-side routes (multipart/form-data):

- `POST /api/files/company-logo`
- `POST /api/files/brand-logo`
- `POST /api/files/participation-logo`
- `POST /api/files/participation-material`

Notes:
- routes are server-only and use R2 credentials on the server
- no presigned upload in this phase
- UI upload forms are implemented and wired to these routes:
  - company detail (`/companies/[id]`) for company logos
  - brand detail (`/brands/[id]`) for brand logos
  - participation detail (`/participations/[id]`) for participation logo and materials

### Curl examples

Company logo:

```bash
curl -X POST http://localhost:3000/api/files/company-logo \
  -F "companyId=00000000-0000-0000-0000-000000000000" \
  -F "logoRole=full" \
  -F "file=@./test-logo.png"
```

Brand logo:

```bash
curl -X POST http://localhost:3000/api/files/brand-logo \
  -F "brandId=00000000-0000-0000-0000-000000000000" \
  -F "logoRole=full" \
  -F "file=@./test-logo.png"
```

Brand logo (inverted full):

```bash
curl -X POST http://localhost:3000/api/files/brand-logo \
  -F "brandId=00000000-0000-0000-0000-000000000000" \
  -F "logoRole=full_inverted" \
  -F "file=@./test-logo-white.png"
```

Participation logo:

```bash
curl -X POST http://localhost:3000/api/files/participation-logo \
  -F "participationId=00000000-0000-0000-0000-000000000000" \
  -F "file=@./test-logo.png"
```

Participation material:


### Response shape

Successful response:

```json
{
  "id": "uuid",
  "provider": "cloudflare_r2",
  "bucket": "exhibition-public-assets",
  "storage_path": "organizations/.../file.ext",
  "public_url": "https://.../organizations/.../file.ext",
  "file_category": "logo",
  "status": "approved"
}
```

Error response:

```json
{
  "error": "message"
}
```
Logo Set helpers:

- `loadBrandLogoSet(brandId)`
- canonical display fallback:

  2. `full`

- `primary_logo_file_id` remains a fallback pointer and list-display default.
